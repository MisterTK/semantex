## 4. Tier 1 — v0.3.1 (immediate, ~1-2 days, target: eliminate 7 regressions)

### Item 1 — Adaptive output budgets based on index size
**Goal:** Stop emitting verbose multi-section responses on small repos and stop truncating on large repos.

**Motivation:**
- gin Q1 +122% (run25 gin.json line "question_id":"Q1","arm":"treatment") — gin has 99 files / 2229 chunks; the fixed `god_nodes=10 + 5 communities + 25 boundaries` overview is overkill and triggers the agent to do follow-up exploration.
- platform Q4 +29% — platform has 824 files / 7809 chunks; `ExhaustiveStructural` always uses `max_results=30` which is too much for an enumeration question that wants ~10 hits.
- CopilotKit Q5 +34% — feature_planning on a 21705-chunk repo, the handler doesn't bound result count adequately.

**Files (exclusive write):**
- `crates/semantex-core/src/search/agent.rs` — handlers and new `ArchBudget` helper
- `crates/semantex-core/src/index/architecture.rs` — `build_arch_overview` signature accepts overrides

**Current behavior:**
- `handle_architecture` in `agent.rs` calls `build_arch_overview(store, &db_path)` which hard-codes 10 god_nodes + 5 communities + 25 boundaries.
- `handle_exhaustive_structural` uses `SearchQuery::new(query).max_results(30)`.
- `handle_deep_with_examples` adds up to 3 patterns × 3 examples = 9 inlined hits regardless of result set size.

**Desired behavior:**
Add a helper that derives output budgets from the chunk count:
```rust
struct ArchBudget {
    god_nodes: usize,
    communities: usize,
    boundaries: usize,
    deep_examples_max: usize,
    exhaustive_max: usize,
}

fn budget_for_chunk_count(chunks: usize) -> ArchBudget {
    match chunks {
        0..=2_000   => ArchBudget { god_nodes: 5,  communities: 3, boundaries: 10, deep_examples_max: 1, exhaustive_max: 12 },
        ..=10_000   => ArchBudget { god_nodes: 10, communities: 5, boundaries: 25, deep_examples_max: 3, exhaustive_max: 25 },
        _           => ArchBudget { god_nodes: 15, communities: 8, boundaries: 40, deep_examples_max: 5, exhaustive_max: 40 },
    }
}
```

`build_arch_overview` gains an optional `Option<ArchBudget>` argument (defaults to medium). The three handlers (`handle_architecture`, `handle_exhaustive_structural`, `handle_deep_with_examples`) read the index's chunk_count via `ChunkStore::chunk_count()` (already exists), compute the budget, and pass through.

**Acceptance:**
- Build clean. `cargo test --workspace --lib` passes.
- Unit test: `budget_for_chunk_count(500) == ArchBudget { god_nodes: 5, ... }`, `budget_for_chunk_count(15_000) == ArchBudget { god_nodes: 15, ... }`.
- Benchmark: re-run gin Q1 (cost ~$1, 2 minutes) — TX turns should be ≤25 (was 47), CCB Δ should be ≤ +20% (was +122%).
- Benchmark: re-run platform Q4 — CCB Δ should be ≤ +10% (was +29%).
- Benchmark: re-run CopilotKit Q5 — CCB Δ should be ≤ +15% (was +34%).

### Item 2 — Multi-language classifier disambiguation
**Goal:** Stop over-matching the `Structural` route on generic queries against multi-language repos.

**Motivation:**
- platform Q2 +69% — platform is TS+Dart. Q2 is "How does this project handle errors?". The current classifier matches "handle" / "uses" / "references" in `structural_keywords` too aggressively and routes to `Structural` which then does a graph walk that returns a small focused result — but Q2 wants a comprehensive multi-language synthesis.

**Files (exclusive write):**
- `crates/semantex-core/src/search/agent_classifier.rs` — extend `classify_agent_query`
- `crates/semantex-core/src/index/storage.rs` — add a small helper to detect multi-language indexes

**Current behavior:**
In `agent_classifier.rs::classify_agent_query`, the Structural route fires on any of these keywords: `"callers"`, `"callees"`, `"who calls"`, `"what calls"`, `"called by"`, `"used by"`, `"uses"`, `"depends on"`, `"references"`, `"call graph"`, `"type hierarchy"`, `"imports"`, `"imported by"`. The phrase "How does this project handle errors?" doesn't directly match any of these, so this is _not_ the issue — confirm by running the classifier:
```
cargo test -p semantex-core --lib search::agent_classifier::tests::test_q2_error_handling
```
(add this test if missing.)

Investigation note: the regression may actually be in the Deep handler's behavior on multi-language repos, not the classifier. Subagent should first run the classifier on the exact Q2 wording from `benchmarks/agent_bench.py::QUESTIONS` and confirm which route fires.

**Desired behavior:**
- Add `pub fn detect_languages(db_path: &Path) -> Vec<String>` to `storage.rs` that reads distinct `language_name` from the `chunks` table (or wherever language is stored).
- If 2+ languages are present AND the query contains "handle" / "deal with" / "support" without a specific symbol → prefer `Deep` over `Structural` (force-override).
- Add a test: `classify_for_repo("How does this project handle errors?", &platform_db_path) == AgentRoute::Deep` when platform has both TS and Dart.

**Acceptance:**
- Build clean.
- New unit test passes.
- Benchmark: platform Q2 TX CCB Δ should be ≤ +20% (was +69%).

### Item 3 — Trim `handle_deep_with_examples` adaptively
**Goal:** Don't pad short responses with excessive pattern exemplars.

**Motivation:**
- flask Q3 +54% — flask Q3 ("explain the most complex algorithm or data transformation") on a 1616-chunk repo. The deep response is already comprehensive; adding 3 patterns × 3 examples adds verbose tokens for a narrow scope.

**Files (exclusive write):**
- `crates/semantex-core/src/search/agent.rs` — `handle_deep_with_examples`

**Current behavior:**
```rust
// extracts up to 3 pattern names from the deep response,
// then queries up to 3 exemplars per pattern → up to 9 inlined hits
```

**Desired behavior:**
Cap pattern hits by `deep_examples_max` from Item 1's `ArchBudget`. Implementation: pass the chunk_count via the budget, and:
- Small repo (chunks < 2000): cap at 1 pattern × 2 examples = 2 hits
- Medium (2-10k): cap at 2 patterns × 3 examples = 6 hits  
- Large: keep current 3 patterns × 3 examples = 9 hits

Also: if `deep_result.result_count < 5`, skip pattern enrichment entirely — the answer is already focused.

**Acceptance:**
- Build clean.
- Unit test: `handle_deep_with_examples` on a synthetic deep result with 3 sources skips pattern enrichment.
- Benchmark: flask Q3 TX CCB Δ should be ≤ +20% (was +54%).

### Item 6 — Confidence-driven follow-up suggestions
**Goal:** When confidence is Ambiguous, suggest a refined query so the agent picks correctly on first retry.

**Motivation:**
Currently every result has `Confidence::{Extracted, Inferred, Ambiguous}` (W1 from v0.3). The Ambiguous case happens when multiple results are within 5% score; the agent sees the top result and either accepts or runs another search. If we surface the runners-up by name, the agent can refine without a new full search.

**Files (exclusive write):**
- `crates/semantex-core/src/server/protocol.rs` — `SearchResponse` / `AgentResponse` add `disambiguation: Option<Vec<DisambigSuggestion>>`
- `crates/semantex-core/src/search/agent_formatter.rs` — render the suggestions inline at the end of the response
- `crates/semantex-core/src/search/agent.rs` — populate when top result is Ambiguous

**Desired behavior:**
When the top SearchResult has `confidence: Ambiguous`, populate `disambiguation` with up to 3 candidate refinements: the next 3 results' top-level chunk names. Format:
```
[uncertainty: this query matches 4 distinct concepts. Refine with:]
- "userAuthHandler" (auth/users.rs:42)
- "tokenAuthHandler" (auth/tokens.rs:18)
- "sessionAuth" (sessions/handler.rs:107)
```

**Acceptance:**
- Test: synthetic SearchResult with Ambiguous confidence produces 3 disambiguation entries.
- Test: with Extracted confidence, disambiguation is None.

### Item 7 — Adaptive structural walk depth
**Goal:** Auto-collapse oversized caller/callee lists; auto-expand undersized ones.

**Motivation:**
`semantex_callers` returns depth=1 by default. For a well-used utility function (`format_string`, `log_error`), depth=1 can return 100+ direct callers. The agent immediately filters — but those 100 results consume context.

**Files (exclusive write):**
- `crates/semantex-core/src/search/agent.rs` — `handle_structural`
- `crates/semantex-core/src/server/handler.rs` — `graph_walk_from_store` (signature change)

**Current behavior:**
`graph_walk_from_store(store, symbol)` returns all callers + callees + type_refs + hierarchy at depth 1.

**Desired behavior:**
After the initial walk:
- If `callers.len() > 50`: collapse by file/module, return summary like "78 callers across 12 files: src/auth/* (24), src/api/* (18), ..."
- If `callers.len() < 5`: re-walk with depth=2, merge results, deduplicate.

**Acceptance:**
- Test on a synthetic graph with 100 callers — output is collapsed by module.
- Test on a synthetic graph with 2 callers — output expands to depth=2.

### Item 8 — Cross-source dedup in deep responses
**Goal:** When deep returns 20 sources, identify duplicates (same chunk in different presentations) and merge them.

**Motivation:**
Deep responses sometimes include the same logical chunk via:
- (a) Direct match
- (b) Match on its NL annotation (E3)
- (c) Match on a containing structured_meta query

These are 3 sources to the agent but 1 piece of code. Stronger dedup means smaller responses without information loss.

**Files (exclusive write):**
- `crates/semantex-core/src/search/deep.rs` — dedup pass before formatting

**Desired behavior:**
Compare results by `(file_path, start_line, end_line)` tuple. Merge duplicates: keep the one with highest score; aggregate annotations from the others into a `also_matched_via: Vec<String>` field.

**Acceptance:**
- Test: synthetic deep result with 5 entries (3 duplicates) collapses to 3 entries.
- Benchmark: any deep-search-heavy question shows smaller response size with same or higher answer accuracy.

**See Spec L** for the replacement design (Phase 1: genai adapter / Phase 2: SubscriptionCli backend). Do NOT implement Item 9 against this spec; implement against Spec L.

### Coordination rules (carry-over from v0.3 spec)
- File conflict rule: if two workstreams need to touch the same file, this spec must state which lines/functions belong to which. No overlapping write claims.
- Coordination request: if a subagent finds it needs to touch a non-owned file, it opens `coordination_request.md` in the repo root rather than editing.
- Worktree discipline: each subagent works in `isolation: worktree` and commits its work in the worktree before returning. Controller merges branches at integration barriers.
- Always reset the integration checkout to a clean state before merging branches in (subagents have been observed leaking writes into the parent checkout; see `~/.claude/projects/.../memory/parallel-subagent-worktree-leakage.md`).
